# Building thinkur

## The macOS Provenance Problem

macOS Sequoia (and later) applies `com.apple.macl` and `com.apple.provenance` extended attributes to files when build tools (xcodebuild, swift build, codesign, etc.) access them inside certain directories ‚Äî notably `~/Downloads`.

These xattrs are **enforced at the kernel level**:
- `ls`, `cat`, `git`, editors, and all CLI tools get `EPERM: Operation not permitted`
- `sudo xattr -cr` also fails ‚Äî the kernel blocks even root
- The **only fix** is to delete the folder in Finder (which bypasses the kernel lock) and re-clone

This has happened multiple times on this project. It is not recoverable without re-cloning.

## Safe Build Workflow

The source repo lives in `~/Downloads/thinkur` (the working tree for code changes). Builds must happen elsewhere.

### For debug builds (Xcode)

Use Xcode's built-in build (Cmd+R). Xcode writes to `DerivedData`, not the source tree. The scheme's post-build action runs `scripts/install-dev-app.sh`, which copies the built app to `~/Applications/thinkur Dev.app`.

The Debug build produces **thinkur Dev** with bundle ID `com.jyo.thinkur.dev`, so it runs side-by-side with the Release app without TCC permission conflicts.

### For release builds

The `scripts/build-dmg.sh` script handles this automatically:

1. Runs `xcodegen generate` in the source tree (safe ‚Äî only writes `.xcodeproj`)
2. Copies the project to `/tmp/thinkur-release-XXXX` via `rsync`
3. Strips any existing xattrs from the temp copy
4. Runs `xcodebuild archive`, export, codesign, notarize ‚Äî all in `/tmp`
5. Copies only the final DMG back to `build/`
6. Cleans up the temp directory on exit

This means **no build tool ever touches the source tree**, avoiding the provenance issue entirely.

### What NOT to do

```bash
# NEVER do any of these in ~/Downloads/thinkur:
xcodebuild ...          # will add provenance xattrs
swift build             # will add provenance xattrs
swift test              # will add provenance xattrs ‚Äî use only if you accept the risk
xcodebuild test ...     # will add provenance xattrs
```

If you need to run `swift test` for quick iteration, understand that it may trigger the provenance lock. The `scripts/build-dmg.sh` approach (copy to /tmp first) is the only guaranteed safe method.

### First-time setup

Run `scripts/bootstrap-release-tools.sh` to download and cache Sparkle's `generate_appcast` and `sign_update` tools to `~/.cache/thinkur/tools/`. This is idempotent ‚Äî it skips if already cached.

### Version numbers

- `MARKETING_VERSION` ‚Äî user-facing version (e.g., `1.0.2`), shown in the app UI
- `CURRENT_PROJECT_VERSION` ‚Äî build number (e.g., `3`), used by Sparkle for update comparison

Both are set in `project.yml` and bumped by `scripts/bump-version.sh`. Sparkle compares `sparkle:version` (the build number), **not** `sparkle:shortVersionString` (the marketing version). If the build number in the app matches the appcast, Sparkle reports "no update available."

### Signing and notarization

- Signing identity: `Developer ID Application` (Team ID: `YZ9FFMX8QS`)
- Notarization: stored as Keychain profile `thinkur-notarize`
- Entitlements: `Sources/thinkur/Resources/thinkur.entitlements` (audio input, bluetooth, network)

### Sparkle auto-updates

- `SUFeedURL` in Info.plist is `$(THINKUR_SU_FEED_URL)` ‚Äî set to the appcast URL in Release, empty in Debug
- `appcast.xml` is generated by Sparkle's `generate_appcast` tool (cached via `scripts/bootstrap-release-tools.sh`)
- Hosted via GitHub Pages on `jyoutir/thinkur-web` (public/ directory)
- DMGs are attached as GitHub Release assets on `jyoutir/thinkur-web`

### Pre-flight checklist

Run `./scripts/release-preflight.sh` to verify:
- Required tools: `xcodegen`, `create-dmg`, `gh`, `xcrun`
- GitHub auth via `gh auth status`
- `Developer ID Application` signing identity in Keychain
- `thinkur-web` repo is cloned alongside this repo
- Git working tree is clean, on `main` branch
